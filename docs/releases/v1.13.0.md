# v1.13.0 · OAuth 拒绝隔离与批量授权稳定性

本版本由 `aiis2` 整理并发布，适配当前 OAuth consent Server Action 协议，补强大批量授权的预检、指标与熔断边界，并把明确拒绝的账号从普通失败记录升级为可迁移、可审计且只能手动恢复的持久禁用账号池。

## 授权与启动稳定性

- consent 页面按 `submitOAuth2Consent` 精确发现当前部署的 Server Action ID，不再误用同页面的其他 action；必要时只读取同源脚本完成发现。
- 授权请求使用当前 Next.js RSC action 数据结构和 Router State Tree；同时保留回调重定向解析，服务端 action 返回错误时直接展示真实原因。
- 批量重新授权先执行单账号预检，并记录本轮成功、失败、跳过和结束状态；重复的系统性失败达到阈值后熔断剩余任务，避免继续放大无效请求。
- `/api/cpa/refresh-all` 与 `/api/cpa/reauthorize` 保持兼容，经典界面和新版界面调用同一批量授权实现。
- Windows 启动器使用模块方式启动面板并显式设置模块搜索路径；若端口上已经是健康面板，则直接复用并打开页面，避免重复启动实例。

## 主要变更

- OAuth 返回 `Access denied` / `access_denied` 时，按规范化邮箱和 SSO 指纹识别稳定身份，写入 `disabled/accounts.json`，同时把已有 CPA 文件标记为 `disabled`。
- 只有账号级明确拒绝会进入禁用池；429、临时 5xx、网络超时、响应解析错误和其他系统性问题继续按原有失败策略处理。
- 禁用账号不再进入自动 CPA 转换、补转、账号重新授权或批量 OAuth 队列；worker 在出队后还会再次检查，避免排队期间状态变化造成竞态。
- 账号列表、单文件下载、合并 SSO、CPA ZIP、Sub2 ZIP/JSON、grok2api、全部产物 ZIP、失败记录和其他聚合导出统一排除禁用身份。
- CPA JSON 保留 `disabled: true` 作为纵深保护；即使旧文件仍存在，也不会进入 CPA/Sub2 导出。

## 页面与恢复

- 新版“账号”页新增禁用账号池，支持独立分页、数量统计和错误状态展示。
- 页面 API 只返回记录 ID、邮箱、原因、时间和来源等公共字段，不返回密码、Web SSO、Access Token 或 Refresh Token。
- “恢复并重新授权”必须由用户手动确认；恢复后强制进入重新授权队列，不会自动复用旧授权。
- 如果重新授权入队失败，禁用登记会原子回滚；新授权成功提交前，旧 CPA 文件继续保持禁用。
- 一批重新授权候选全部被明确拒绝时，任务会完成禁用收敛而不启动空的批量队列；系统性预检失败仍会停止整批任务并返回明确错误。

## 存储与迁移

- 凭据布局新增 `disabled/` 目录，禁用登记使用跨进程文件锁和同目录临时文件原子替换。
- “迁移并切换”会把禁用登记与 SSO、邮箱 JWT、CPA、OAuth 所有权账本一起迁移；多来源记录按稳定 ID 合并。
- 凭据状态接口新增禁用登记文件计数，但不会返回登记内容。
- 注册和转换成功不会隐式恢复账号；恢复权限只存在于明确的页面操作与对应 JSON POST 接口。

## 升级说明

1. 停止旧面板，保留本地 `config.json`、`data/credentials/` 和 `data/logs/`，更新程序文件后重新运行 `start.bat`。
2. 现有凭据目录会自动创建 `disabled/`；如需修改目录，请使用页面的“迁移并切换”，禁用登记会随其他凭据一起迁移。
3. 对历史日志中的 `Access denied` 不做离线猜测或批量禁用；只有升级后实际 OAuth consent 明确拒绝的身份才会进入禁用池。
4. 在“账号”页检查禁用账号。只有确认账号可重新授权时才点击“恢复并重新授权”，随后等待 CPA 队列完成再导出。

## 安全边界

- 禁用登记包含恢复队列所需的内部身份信息，属于敏感凭据的一部分；不要提交、上传或公开该文件。
- 页面、日志、测试产物和本说明均不包含真实密码、Web SSO、Access Token、Refresh Token 或邮箱管理密钥。
- 本版本阻止当前凭据仓库中的禁用身份再次导出，但无法撤回此前已复制到外部实例的历史 Token；外部实例仍需单独停用对应账号。
