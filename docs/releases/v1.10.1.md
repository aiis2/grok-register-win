# v1.10.1 · Chromium 验证恢复与千轮稳定性验证

本版本由 `aiis2` 整理并发布，修复 Chromium 有头注册在最终资料页长时间停留于 Cloudflare Turnstile 灰色验证区域的问题，并补充 1000 轮、10 并发槽的本地端到端稳定性验证。

## 主要变更

- 将 Turnstile token 读取和控件交互收敛为浏览器后端共用的单次非阻塞操作；Camoufox 继续使用原有帧点击实现，Chromium 使用 DrissionPage 访问嵌套 Shadow DOM。
- Chromium 最终资料页不再只等待 token 或执行 reset，而会按 `1.5s` 节流主动检查并交互；拿到有效 token 后立即回到既有提交状态机。
- 保留原有恢复边界：长时间未就绪只执行一次官方 reset，随后仍无结果则仅重启当前并发槽的浏览器与 Profile；不会无限等待或影响其他 worker。
- Turnstile 调试日志只记录交互方式、是否点击和 token 长度，不记录 token、Cookie、密码或邮箱管理凭据。
- 新增 1000 轮、10 个并发槽的确定性本地压力测试，覆盖任务切片、并行汇总、结果去重、Worker 完成态和日志有界存储。

## 根因

Chromium 路径原本已经包含可工作的 DrissionPage Shadow DOM 交互，但最终资料页等待循环仅在对象具有 `click_turnstile()` 时主动调用交互。该方法属于 Camoufox 包装层，Chromium 没有这个属性，因此实际流程会反复观察到空 token，经过 reset 后更换 Profile，却从未调用已有的 Chromium 交互路径。

现在资料页统一调用浏览器无关的单次交互函数，并由外层状态机控制节流、reset、重启和硬超时。

## 验证

- 真实 Chromium 隐藏有头单轮验证：进入资料页后约 4 秒完成 Shadow DOM 点击，约 6 秒得到有效 token；注册、grok.com SSO 获取和 CPA 转换完整成功。
- 完整自动化测试：`381 passed`；Python 编译和 `git diff --check` 通过。
- 1000 轮本地压力测试：10 个并发槽同时启动，1000 个结果全部且仅记录一次，最终为成功 1000、失败 0；后端普通日志与 SSE 事件缓冲均稳定限制在 2000 条。
- Playwright 加载 1000 轮、10 Worker、2000 条历史日志：页面约 `0.88s` 就绪，默认只绘制 300 行；追加 1000 条实时日志约 `0.54s`，没有 Long Task，DOM 仍为 300 行。
- Playwright 验证加载更早为 600 行、回到最新恢复 300 行、搜索命中、深色主题、390px 无横向溢出，浏览器控制台错误为 0。
- 账号分页在 2557 条现有注册记录上返回 103 页，前后翻页与精确搜索正常。
- 对 1 个现有 SSO 执行真实刷新：入队 1、成功 1、失败 0、队列归零，恰好 1 个 CPA 文件原子更新且无新增或删除。
- Sub2API JSON 与 ZIP 均完成内存解析；两种产物都包含 2510 个结构有效的 Grok OAuth 账号，ZIP 的逐账号文件数与 `all.json` 一致。

1000 轮压力项使用确定性本地 worker，不会批量创建外部平台账号；真实网络链路使用单账号验证，避免把发布验证本身变成对第三方服务的大规模请求。

## 兼容性与升级

- 现代面板、经典界面、注册与日志合并布局、账号分页、邮箱配置、凭据迁移、SSO 刷新及 CPA/Sub2API 导出协议不变。
- 现有 `config.json`、`data/credentials/` 与 `data/logs/` 可直接复用。
- 升级后停止旧面板并重新运行 `start.bat`，确保新的注册 worker 载入本版本脚本。
