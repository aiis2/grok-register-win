# v1.6.0 · Windows 隐藏有头浏览器

本版本由 `aiis2` 整理并发布，重点解决并发注册时有头浏览器抢占 Windows 前台、任务栏堆积以及验证组件长时间停滞的问题。

## 主要变更

- 新增 Chromium 窗口模式：面板和 `config.json` 均支持 `hidden`、`minimized`、`visible`；Windows 默认使用 `hidden`。
- 实现方案 A“隐藏有头”：仍启动完整 Chromium 有头实例，不加入任何 `--headless` 参数；通过静默启动、后台 CDP target 与 Windows 原生窗口控制，让注册窗口默认不可见且不出现在任务栏。
- 增加安全降级：若静默 CDP 引导失败，只终止本次刚创建的精确进程树并回退到最小化有头模式，不会切换无头模式；Worker 卡片会显示降级状态。
- 增加 Worker 浏览器归属协议：CLI 上报 worker、generation、PID、HWND、模式和状态，面板拒绝旧代次或不匹配的窗口记录。
- 增加页面手动显隐：运行中的 Worker 卡片可显示或隐藏自己的浏览器；显示一个 worker 前先隐藏其他已登记窗口，只有用户显式点击“显示浏览器”时才允许激活目标窗口。
- 加固控制边界：所有 Win32 操作在执行前校验 HWND 仍属于捕获的 PID，执行后再次校验 generation/PID/HWND；从不按进程名或窗口标题扫描控制用户自己的 Chrome。
- 保持任务级生命周期：同一 worker 在当前批次中复用健康浏览器，任务完成、停止或失败后立即退出，不跨下一次注册任务保留常驻进程。
- 改进 Turnstile 卡死恢复：识别仅剩 64px 灰色占位、组件未初始化或 token 长时间为空的状态；执行一次有界 reset，仍未恢复时仅重启当前 worker 的浏览器和独立 Profile，避免耗完整轮超时。
- 改进原有最小化路径：窗口启动和最小化操作不主动调用前台激活，并记录精确归属，降低多并发启动时的桌面干扰。

## 配置与兼容性

```json
{
  "browser_engine": "chromium",
  "browser_window_mode": "hidden"
}
```

- `hidden` 仅在 Windows 上生效；其他平台会规范化为 `visible`。
- `browser_window_mode` 仅影响 Chromium；选择 Camoufox 时仍按既有无头模式运行。
- 旧配置没有该字段时，Windows 自动采用 `hidden`，无需迁移。
- 隐藏窗口仍保留正常有头 User-Agent、渲染和验证组件能力；它不是 headless，也不是截图式远程浏览器。

## 安全与资源行为

- 浏览器窗口仅能由产生它的当前 worker 记录控制。
- 窗口句柄被关闭、复用或浏览器重启后，控制 API 返回冲突错误，不会扩大搜索范围。
- 一个并发槽最多持有一个 CLI 子进程和一个浏览器根进程；Chrome 自身的 renderer、GPU、network 子进程仍属于该根进程树。
- 停止、超时和静默引导失败均按已捕获 PID 清理，不会结束系统中同名的其他浏览器。
- 配置文件、日志、测试和 Release 不包含 Freemail、SMTP、账号或其他真实凭据。

## 验证范围

发布前执行完整 pytest、Python 编译、HTML/JavaScript 检查、面板桌面与窄屏渲染，以及 Windows 原生运行时验证。运行时验证覆盖：隐藏启动过程中目标浏览器不成为前台窗口、不出现任务栏入口；手动显示/隐藏复用同一 PID 与 HWND；页面状态保持；User-Agent 不包含 `HeadlessChrome`；任务结束后精确进程树清零。

注册成功率仍受代理出口、xAI/Cloudflare 风控、邮箱域名信誉和第三方服务状态影响。隐藏窗口模式只解决桌面焦点与任务栏干扰，不绕过验证码或平台风控。
