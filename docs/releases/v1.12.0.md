# v1.12.0 · OAuth 单实例归属与账号重新授权

本版本由 `aiis2` 整理并发布，解决同一身份的 refresh token 被旧凭据重复导入、被多个消费实例同时使用，以及 token rotation 后重新导入失效授权的问题。CPA 网络流水线、现代面板和经典界面继续保持兼容。

## 主要变更

- 新增“重新生成账号授权”：使用已有 Web SSO 重新执行 OAuth authorization-code exchange，生成新的 access token 与 refresh token；该操作不会生成新账号，也不会拿旧 refresh token 做刷新。
- 授权队列优先按稳定 subject 去重，缺失时回退到规范化邮箱或 SSO 指纹；同一身份一次任务只处理一次。
- 每次成功授权写入独立的授权 ID、授权代次和授权时间，并通过同目录临时文件原子替换 CPA；失败时保留旧文件。
- 新增 OAuth 单实例归属账本，仅保存 identity/token 的 SHA-256 指纹、目标实例和授权代次，不保存邮箱、密码、Web SSO 或明文 Token。
- CPA ZIP、Sub2 ZIP 与 Sub2 JSON 导出统一执行只读预检、POST 原子认领和一次性票据下载；票据 60 秒过期、只能使用一次且绑定具体导出产物。
- 同一 refresh token 已归属其他身份或目标实例时导出失败关闭；迁移到新目标实例必须先停用旧实例账号并明确确认。
- 升级前缺少授权代次的旧 CPA 不会被直接认领或导出，必须先执行一次“重新生成账号授权”。

## 并发与工作区安全

- OAuth worker 从 authorization-code exchange 到原子提交持续持有跨进程凭据租约，避免其他面板进程同时生成、迁移、认领或导出同一工作区。
- 同一面板内 `1–4` 个 worker 共享本进程租约，继续保留网络阶段并行能力；CPA 文件和索引仍由单提交器串行落盘。
- 导入、迁移和目录切换会推进持久化 workspace epoch，使尚未开始兑换的旧任务安全失效；普通 CPA 提交不会误杀同批后续任务。
- 导入激活、SHA-256 校验或配置保存失败时，会在重新放回旧队列前恢复 durable epoch，避免有效任务被误判为 stale 后静默丢弃。
- 凭据迁移会按规范化路径锁定新旧归属账本；非冲突账本合并到唯一标准文件，归属冲突时保留原配置和来源文件。

## 界面与操作

- 新版“凭据 → CPA 转换”和经典界面均提供“重新生成账号授权”入口。
- 配置页新增 `OAuth 目标实例`，用于唯一标识实际消费 CPA/Sub2 凭据的一套部署。
- 导出前会展示旧授权、缺失 refresh token、实例冲突和目标迁移风险，不会在 GET 请求中隐式改变所有权。
- README 和示例配置已补充单实例归属、重新授权、目录迁移及安全升级步骤。

## 升级步骤

1. 停止旧版面板进程，并停用旧 CPA/Sub2API 实例中的对应账号。
2. 保留本地 `config.json`、`data/credentials/` 与 `data/logs/`，更新程序文件后重新运行 `start.bat`。
3. 在“凭据 → CPA 转换”执行“重新生成账号授权”，等待队列归零。
4. 保存唯一且稳定的 `OAuth 目标实例`，再重新导出 CPA/Sub2 凭据并只导入该实例。

## 安全边界

- 本版本可以阻止共享同一凭据目录且运行当前版本的程序重复生成或分发同一身份授权。
- 已经手工复制到其他目录、旧程序或外部实例的历史 Token 无法自动撤回；切换目标实例前仍必须先停用旧实例账号。
- 状态 API、页面、日志、测试结果和本说明均不包含 Web SSO、Access Token、Refresh Token、密码或邮箱管理密钥。
