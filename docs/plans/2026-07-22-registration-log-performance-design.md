# 注册与日志合并及大日志性能优化设计

## 背景与根因

现代面板当前将“注册”和“日志”放在两个独立主导航分区。日志缓冲区在服务端和浏览器端均最多保留 2000 条，但浏览器收到每一条 SSE 事件时都会同步执行一次 `renderLogs()`：重新过滤全部日志、重新创建全部行节点、替换 `#logs-output` 的全部子节点，并读取 `scrollHeight` 后滚动到底部。

使用 Playwright 向隔离面板预装 2000 条日志后，从注册页点击日志页产生了约 49.3 秒的单次 Long Task，最终创建 2000 个 `.log-line` 节点。卡顿来自 SSE 历史回放期间反复执行逐渐扩大的整表重建，复杂度接近 O(n²)，不是 Flask 日志缓冲或 SSE 协议本身阻塞。

## 目标

1. 将现代面板的“注册”和“日志”合并为一个“注册与日志”主导航项。
2. 页面上方保留注册参数、执行状态和 Worker 控制，下方提供全宽实时日志控制台。
3. 浏览器仍保留最近 2000 条日志，但默认最多渲染最近 300 条匹配记录。
4. SSE 历史回放和实时突发按批次更新，避免逐条整表重建和逐条强制滚动。
5. 保留暂停、自动滚动、级别筛选、搜索、重连、轮询回退与本地清空能力。
6. 兼容旧的 `#logs` 深链，并保持经典界面与后端 API 不变。

## 方案选择

采用“有界可视窗口 + 批量增量渲染”，而不是只做节流或完整虚拟列表。

- 只做节流改动较小，但筛选和恢复显示时仍可能一次创建 2000 个节点，无法建立稳定的 DOM 上限。
- 完整虚拟列表可以浏览全部记录，但日志行允许换行、行高不固定；自行实现动态高度虚拟化会显著增加滚动、焦点和可访问性复杂度。
- 有界窗口在内存中保留全部 2000 条，默认只呈现最近 300 条，并允许每次再加载 300 条更早记录。它能稳定控制 DOM 规模，同时不丢失当前会话的日志历史。

## 页面结构

- 主导航删除独立“日志”项，将“注册”改名为“注册与日志”。
- `#section-register` 中依次放置任务参数与状态、Worker 列表、实时日志区域。
- 日志区域使用 `id="registration-log-console"` 作为兼容定位锚点，保留现有日志控件 ID，避免破坏事件绑定和自动化选择器。
- `#register` 是规范 hash；访问旧 `#logs` 时进入注册分区，并在首轮日志渲染后滚动到日志区域。
- 概览中的“查看日志”继续使用 `#logs`，由兼容映射直接定位日志，而“开始一轮注册”使用 `#register` 定位页面顶部。

## 日志状态与渲染

浏览器仍保存：

- `lines`：SSE 模式最近 2000 条去重事件；
- `fallbackLines`：轮询模式最近 2000 条日志；
- 当前级别、搜索词、暂停和自动滚动偏好；
- `visibleLimit`：默认 300，每次“加载更早”增加 300，最大不超过匹配总数；
- 待渲染计时器、动画帧和是否需要完整同步的标记。

渲染分为两条路径：

1. **完整同步**：首次进入、切换 SSE/轮询、筛选、搜索、加载更早、清空和暂停恢复时执行。先过滤全部内存记录，再只截取末尾 `visibleLimit` 条创建 DOM。
2. **实时批次**：SSE 事件只写入内存并安排一次刷新。刷新间隔约 100ms，并通过 `requestAnimationFrame` 对齐绘制；一个批次只触发一次完整同步。默认窗口最多 300 行，因此即使历史回放包含 2000 个事件，也只会执行少量有界重绘。

本轮不引入 npm、前端框架或第三方虚拟列表。批次刷新选择有界整窗同步，而不是复杂的逐节点差量算法：300 行的重建成本稳定、筛选状态一致，代码更容易验证；关键是消除每个 SSE 事件触发一次 1–2000 行重建的二次复杂度。

搜索输入使用约 180ms 防抖；级别变化立即同步。日志计数显示“显示 N / 匹配 M / 保留 T”，当 `M > N` 时展示“加载更早”，加载完毕或没有日志时禁用。恢复“最近 300 条”用于快速回到实时窗口。

自动滚动每个批次最多触发一次。暂停只冻结 DOM，SSE 仍继续接收并更新计数；恢复时一次性同步最新窗口。切换到其他主分区时连接可继续保持，但不执行日志 DOM 渲染；重新进入时同步一次。

## 兼容性与安全

- `/api/logs/stream`、`/api/job/status`、序号去重和轮询回退契约不变。
- 经典界面不修改。
- 日志仍只保存于进程内和当前浏览器内，不新增持久化或下载接口。
- 不把密码、SSO、Access Token、Refresh Token、邮箱密钥或真实测试配置写入模板、测试、截图、Release 或 Git。
- `#logs` 和已保存的 `panel-v2-section=logs` 自动映射为 `register`，防止升级后落回概览。

## 验证门槛

自动化测试必须覆盖：

- 导航只有一个“注册与日志”项，日志控制台位于 `section-register` 内；
- `#logs` 兼容映射存在；
- 可视窗口默认上限为 300，加载更早按 300 递增；
- SSE 回调不直接调用同步日志渲染，而是进入批量调度；
- 搜索具有防抖，暂停、筛选、重连和清空保持原语义；
- 全量 pytest、JavaScript 语法检查和 Python compileall 通过。

Playwright 使用 2000 条历史日志和额外突发日志验证：

- 从 `#register` 进入合并页后日志完成接收，默认 DOM 行数不超过 300；
- 点击到日志就绪约 1 秒内完成，最长 Long Task 低于 200ms；
- 页面在日志接收期间仍能响应主题切换、搜索、筛选、暂停和恢复；
- “加载更早”和“回到最新”计数、按钮状态正确；
- 1440px 与 390px 视口无页面级水平溢出，控制台无非预期错误。
